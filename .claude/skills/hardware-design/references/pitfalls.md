# エラーにならず結果が不正になるパターン

build123d は、間違った操作でも例外を出さずに処理を続けることがある。
実行が成功したことは、形状が正しいことを意味しない。

## 1. 効かないブール減算（最重要）

対象に当たらない位置で減算しても、例外は出ない。体積も面数も変わらない。

```python
box = Box(40, 30, 10)            # volume 12000.0, faces 6
missed = box - Pos(500, 0, 0) * Cylinder(radius=3, height=20)
# 例外なし。volume 12000.0 のまま、faces 6 のまま
```

穴が開いていないのに、コードは正常に動いたように見える。

**対策:** ブール減算のたびに体積が減ったことを確認する。

```python
from hwlib.verify import assert_cut

before = part
part = part - hole
assert_cut(before, part, name="M3 バカ穴")
```

原因の多くは、位置・向き・サイズの指定ミス。特に切り欠く側の高さが足りず、
壁を貫通していないケースが多い。切り欠く側は必ず壁より長くする。

## 2. 座標が同一平面上にあるとき、フィーチャが融合しない

追加する形状の面が既存の面とちょうど一致していると、確実には結合されない。
少しめり込ませる（1 mm 程度）か、面をきれいに分ける。

## 3. `is_valid` はプロパティであってメソッドではない

```python
part.is_valid()   # TypeError: 'bool' object is not callable
part.is_valid     # True / False
```

## 4. `Plane(origin, z_dir=...)` は x 軸の向きが定まらない

`z_dir` だけを渡すと x 軸が自動で決まるため、幅と高さがどの軸に対応するか不定になる。
コネクタ開口を作る側と確認する側で解釈がずれ、**開口が 90 度回る。**

**対策:** `hwlib.verify.face_plane(origin, face)` を使う。x 軸の向きを固定してある。

## 5. アセンブリで `.move()` の絶対座標を使う

パラメータを変えた瞬間に位置関係が崩れる。`RigidJoint` などの Joint を使うか、
本リポジトリのように「基準位置からの相対配置」をパラメータで表現する。

## 6. 干渉していない `&` は例外ではなく空の Compound を返す

```python
(a & b).volume   # 重なっていなければ 0。例外は出ない
```

これ自体は扱いやすいが、「例外が出ないから干渉していない」ではなく、
**体積を見て判断する**こと。

## 7. コネクタ開口の確認が、壁に届かず素通りする

開口の有無を調べるプローブが短いと、壁まで到達せず「干渉なし = 開口あり」と誤判定する。
`hwlib.verify.assert_connector_access()` は外装の外まで届く長さを自動計算する。
自前で確認するときは同じ注意が必要。

## 8. 収まっていることを、組み立てられることと取り違える

干渉チェックが通っても、次のいずれかで組み立て不能になる。

- ドライバーが入らない → `assert_tool_access()`
- 部品を入れる経路がふさがっている → `assert_insertable()`
- 部品が固定されていない（箱の中で浮いている） → `bom.yaml` の `retention` を必ず書く

干渉チェックは「浮いている部品」を素通りさせる。収まることと固定されることは別物。

## 9. 3 桁の hex カラーは pyvista で使えない

`#334` はエラーになる。`#333344` のように 6 桁で書く。
