# ハードウェア設計

build123d でハードウェアを設計する。目的は 2 つ。

1. **必要な部品が部品表にすべて含まれていること**
2. **3D プリントした外装にすべての部品が収まり、ネジで実際に組み立てられること**

設計は次の 5 段階で進める。**段階を飛ばさない。**

| 段階 | 内容 | 出力 |
|---|---|---|
| 1 | ヒアリング | `projects/<name>/brief.md` |
| 2 | 部品の洗い出し | `bom.yaml`（寸法欄は空） |
| 3 | 寸法調査 | `bom.yaml`（確定） |
| 4 | CAD 設計 | `model.py` |
| 5 | 検証 | `test_model.py`、PNG |

## 段階 1: ヒアリング

`AskUserQuestion` で次を確定させる。曖昧なまま進めない。

- ハードウェアの目的、使用環境（据置か可搬か、屋内か屋外か）
- 大まかな形状、外形サイズの上限
- 使いたい部品で決まっているもの（型番レベルで）
- 外部インターフェース（電源、USB、表示、操作、レンズ）

結果を `projects/<name>/brief.md` に残す。

## 段階 2: 部品の洗い出し

必要な部品をすべて列挙する。`references/bom.md` のカテゴリ表を使い、**各カテゴリについて部品を挙げるか、「不要」の理由を `excluded` に書く**。黙って飛ばさない。

配線とコネクタの挿抜空間は忘れやすい。ケーブルが占める体積も部品として扱う。

## 段階 3: 寸法調査

全部品の幅・奥行き・高さ（mm）とネジ穴位置を確定させる。

1. 型番をもとに `WebSearch` / `WebFetch` でメーカーの機械図面・データシートを調べる
2. 得た値を `source`（出典 URL）とともに `bom.yaml` に書く
3. **ネットで確定できない場合は、推測せずユーザにノギスでの実測を依頼する。** どこを測るかを具体的に示す

**推測値で設計を始めない。** 確認できなかった項目は書かない（例: 取付穴の位置が不明なら `mount_holes` を空にし、その旨を `note` に残す）。

`bom.yaml` は `hwlib.bom.load_bom()` で読む。寸法・出典・固定方法・カテゴリの網羅性に不備があればエラーになり、段階 4 に進めない。

## 段階 4: CAD 設計

1. `bom.yaml` から部品のモック（外形の直方体）を生成し、配置する
2. モックを囲む形で外装を作る。ボス、リブ、開口部、蓋を設ける
3. 締結はタッピングネジ（樹脂直締め）。ボスは `hwlib.features.tapping_boss()` で作る
4. パラメータは冒頭の dataclass にまとめる。**モデル本体に数値を直接書かない**

**収まることと固定されることは別物。** すべての部品について、何で固定するか（ネジ、ポケット、蓋で押さえる、両面テープ）を `bom.yaml` の `retention` に書く。書かないとエラーになる。

詳細は `references/design-rules.md`（肉厚、クリアランス、ボス寸法）と `references/api.md`（build123d の書き方）。

## 段階 5: 検証

**順序を守る。数値検証 → 画像確認 → 人間の確認。逆順にしない。**

build123d のブール減算は、対象に当たらなくても例外を出さない。**体積も面数も変わらないまま処理が成功する。**
レンダリングもこの不正な形状に対して成功するため、画像だけを見ると誤りを見逃す。

### 1. 数値検証

```bash
uv run pytest
```

`hwlib.verify` の関数で以下を確認する（詳細は `references/assembly.md`）。

- `assert_cut()` — **ブール演算のたびに呼ぶ。** 効かなかった穴あけを検出する
- `assert_all_parts_placed()` — BOM の部品がすべて配置されているか
- `assert_contained()` — 各部品が外装の内側に収まっているか
- `assert_no_interference()` — 部品同士、部品とボス・壁が干渉していないか
- `assert_tool_access()` — **ドライバーが届くか**
- `assert_insertable()` — **部品を入れられるか**（収まっていても経路がふさがれていれば組み立てられない）
- `assert_connector_access()` — コネクタの位置に開口があるか
- `assert_boss_screw_fit()` — 下穴径・ねじ込み深さ・ネジ長が成立するか
- `assert_watertight()` — 3D プリントできるメッシュか

### 2. 画像確認

```bash
uv run python -m projects.<name>.model --render
```

生成された PNG を **Read ツールで開いて自分で確認する**。生成しただけで完了としない。

数値検証は「寸法が合っているか」しか見ない。「意図した形か」は画像で確認する。

### 3. 人間の確認

```bash
uv run python -m projects.<name>.model --show
```

ocp_vscode のビューアに送る。**Claude が見ているのはテッセレーション済みメッシュの PNG であり、細かいフィレットやわずかな干渉は判別できない。** この限界を伝えたうえで報告する。「完璧です」と断言しない。

ビューアがロゴ（デフォルト表示）のまま変わらない場合は、**ブラウザを接続してから送信する**という順序を守る。スタンドアロンサーバはモデルを保存・再送しないため、送信の瞬間にブラウザが登録されていないとモデルは捨てられる。ブラウザのタブを F5 で再読み込みしてから `--show` を実行し直す。

## 破綻していた場合

まず修正を試みる。修正が難しい場合（部品が物理的に収まらない、寸法の前提が間違っている）はユーザに問い合わせる。**破綻を残したまま完了報告しない。**
