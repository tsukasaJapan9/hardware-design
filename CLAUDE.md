# このリポジトリの約束事

## 文章は日本語で書く

**これから新しく作るファイルは、日本語で書く。**

対象は、README、スキル定義、コード中のコメントと docstring、検証器が出力するメッセージ、
コミットメッセージ、PR 本文。要するに人間が読むテキストすべてだ。

**すでにある英語のファイルは、英語のままにする。** 既存ファイルを編集するときも、そのファイルの
言語に合わせる。日本語と英語が1つのファイルの中で混ざるほうが、リポジトリ全体で言語が
分かれているより読みにくい。

### 文体

**「だ・である」調**で書く。「です・ます」調は使わない。

### 簡潔かつ明確に書く

1文に1つのことだけを言う。書いたあと、削れる語がないか読み返す。

削るもの:

- **言い換えによる繰り返し。** 同じことを別の言葉でもう一度言わない
- **前置きと断り書き。** 「ここで注意すべきは」「以下に述べるように」は本文ではない
- **中身のない言い回し。** 「〜という点に留意が必要である」は、何も言っていない
- **修辞と比喩。** 気の利いた一文より、正確な一文を選ぶ
- **読者の行動を変えない情報**

残すもの: 事実と、その根拠。とくに**なぜそうなのか**は必ず書く。「最小肉厚は 1.2mm」だけでは
不十分で、「外周3本ぶん。2本だとあいだにインフィルが入らず割れる」まで書く。理由のない
規則は、状況が変わったときに誰も判断できない。

簡潔さと明確さがぶつかったら、明確さを取る。読み返させたり質問させたりしたら、短く書いて
節約した分は消える。

### 訳すのではなく、日本語で書く

英語で下書きしてから訳すと、英語の語順と修辞構造が残り、日本語として読めないものができる。
「推測のネガはスクラップだ」のような英語の言葉遊びを直訳したり、関係詞節の入れ子をそのまま
長い一文にしたりする。実際にそれをやって、全部書き直した。

内容だけを持って、白紙から日本語で書くこと。英語版を横に置かない。

### 技術用語

定着した日本語の技術用語を使う。造語や直訳をしない。

**初出時は「日本語（English）」と書き、英語を併記する。** 2回目以降は日本語だけでよい。
併記するのは、英語の一次資料（build123d のドキュメント、OCC の API、スライサーの設定項目、
ネジやベアリングの規格）を引くときに、どの語で検索すればよいかが分かるようにするためだ。

```
最小肉厚（wall thickness）は 1.2mm とする。外周3本ぶんで、2本だと割れる。
ベアリングは圧入（press fit）で外輪（outer race）だけを押す。
```

| 日本語 | 英語 |
|---|---|
| 圧入 | press fit |
| すきま | clearance |
| 干渉 | interference |
| 公差 | tolerance |
| オーバーハング | overhang |
| ブリッジ | bridge |
| フィレット | fillet |
| 面取り | chamfer |
| 肉厚 | wall thickness |
| ボス | boss |
| 座ぐり | counterbore |
| 皿もみ | countersink |
| 熱圧入インサート | heat-set insert |
| 内輪 / 外輪 | inner race / outer race |
| 象足 | elephant foot |
| 造形板 | build plate |
| 造形範囲 | build volume |
| 多様体 | manifold |

日本語の定訳が無いもの、あるいは英語のほうが通りが良いものは、英語のまま使ってよい
（keepout、footprint、datum など）。この場合は併記しない。

迷ったら、その分野の技術者が実際に口にする語を選ぶ。

**表に無い技術用語を使ったら、その場で表に追加する。** 訳語を決めるのは、その語を最初に
使うときだ。あとでまとめてやろうとすると、同じ語が場所によって違う訳になり、検索もできなく
なる。追加はその変更と同じコミットに含める。

表にある語の訳が適切でないと判断したら、表を直し、**既存の日本語ファイル全体を新しい訳語に
揃える**。表と本文が食い違っている状態を残さない。

### 英語のまま残すもの

コードの識別子。変数名、関数名、クラス名、辞書のキー、`Mode.SUBTRACT` のようなライブラリ由来の
語は英語のままにする。

## git

新しい実装を始める前に、必ず main に戻ってから作業ブランチを切る。

```bash
git checkout main && git pull
git checkout -b feature/<name>
```

## 検証

コードを変更したら、コミット前に必ず通す。

```bash
uv run pytest                                 # 検証器は答えが既知のテストで固定してある
uv run python -m examples.parts_box           # 3つの設計がすべて通ることを確認する
uv run python -m examples.motor_mount
uv run python -m examples.roller_bracket
```

検証器のメッセージ文言はテストが assert している。出力を変えたら、テストも一緒に直すこと。
